# Doctor CRM - Gemini 3.1 開發計畫 (Gemini 3.1 Plan)

**日期**：2026-02-23  
**基於**：現有 `CRMplan.md` 與 `progress.md` 的分析  
**目前狀態回顧**：Phase 1-3 後端模組（包含患者、員工、療程模板、療程會話、PPF 分潤引擎）與基礎 API 已完善，並具備 A 級代碼質量與 400+ 個測試通過。前端基礎架構（Vue 3 + TypeScript + Naive UI）與核心頁面已初步完成，並已進入功能擴展階段。

本計畫旨在透過 Gemini 3.1 的輔助，完成 Phase 4 前端深度整合、Phase 5 系統綜合穩定化測試，並為未來的實名化與 AI 數據分析鋪路。

---

## 🚀 階段 1：Phase 4 前端 UI 深度實作與 UX 優化（近期核心）

目前前端已完成清單與基礎 CRUD，接下來需進入深度的操作體驗優化與複雜業務邏輯介面整合。

### 1.1 PPF 分潤規則與後台配置 UI 完善
- **目標**：完成「療法 × 職位類型 × 分潤比例」的動態配置介面。
- **作法**：
  - 開發並優化 `PPFConfigView.vue`，確保與後端動態表單系統（Percentage, Fixed, Tiered 三種模式）完美對接。
  - 實作「即時預覽/試算」功能：讓財務或管理人員能輸入假定「療程金額」，即時看見各角色的分配結果（包含階梯式抽成的邊界測試）。
- **Gemini 任務**：協助撰寫複雜的表單響應式驗證邏輯與階梯式（Tiered）模塊增刪組件，確保狀態更新即時無延遲。

### 1.2 療程會話（Session）與排程視圖整合
- **目標**：提升治療師與醫師排程與查看療程任務的工作效率。
- **作法**：
  - 引入日曆組件，將 `TreatmentSession` 結合員工指派（`StaffAssignment`）圖形化顯示於日事曆或週曆上。
  - 實作 Drag & Drop 拖曳更新預定時間（支援連動 API 更新時間點與狀態）。
- **Gemini 任務**：撰寫 Vue 3 複雜日曆組件狀態管理與跨組件通訊，確保在 Naive UI 體系下的流暢體驗。

### 1.3 儀表板（Dashboard）與大數據可視化
- **目標**：針對診所管理者提供直觀營運數據與分潤概覽。
- **作法**：
  - 串接圖表套件（如 ECharts）。
  - 可視化呈現本月療程銷售額、各階段執行完成進度、員工 PPF 總體提成報表等。
- **Gemini 任務**：封裝響應式的 Vue 3 + ECharts 圖表組件，並處理從 API 獲取的原始資料聚合與 dataset 轉換。

## 🛡️ 階段 2：Phase 5 系統穩定性、端到端測試與部署準備（中期）

在核心商業邏輯已具備高單元測試覆蓋率下，需進一步加強端到端（E2E）測試與安全實施。

### 2.1 端到端（E2E）與集成情境測試
- **目標**：確保高價值流程（建立病患 -> 購買療程 -> 執行療程 -> 自動產生與鎖定 PPF -> 提交調整）在真實場景中順利無毛病。
- **作法**：
  - 前后端引入自動化測試框架（例如 Playwright 或 Cypress）。
  - 模擬完整的使用者操作路徑與網路不穩定時的情境處理。
- **Gemini 任務**：自動生成包含各種邊界條件的 E2E 測試腳本，覆蓋「事件驅動（Event-Driven）分潤」的非同步驗證與斷言。

### 2.2 醫療級審計日誌（Audit Log）與權限細化（RBAC）
- **目標**：落實醫療與財務數據防竄改機制，確保操作合規。
- **作法**：
  - 利用 NestJS Interceptor 或 Middleware 實作全域操作攔截器，記錄資料異動的時間、人員與新舊對比（Payload）。
  - 對 Controller 加上更細粒度的角色權限裝飾器（例如醫師僅可更新自身相關的 Session 備註，財務專員才有權限執行 Revenue 鎖定或核准）。
- **Gemini 任務**：分析現有 NestJS 架構，產出最佳的面向切面編程（AOP）Audit Log 解決方案與程式碼。

### 2.3 容器化開發與 CI/CD
- **目標**：實現環境隔離一致性以及平滑迭代部署。
- **作法**：
  - 更新並驗證 `docker-compose.yml`，優化 NestJS 與 Vite Dev Server 的共存調試模式。
  - 編寫建置管線（如 Gitlab CI 或 GitHub Actions）準備生產環境（從 SQLite 無縫切換到 PostgreSQL 等）。
- **Gemini 任務**：產出專為 Node/Nest 生態系優化與安全考量的 Dockerfile、打包縮減構建體積腳本。

## 🤖 階段 3：未來的智能 CRM 與 AI 輔助（長期）

藉由 Doctor Toolbox 本身的擴展定位，開始整合強大的 AI 處理能力。

### 3.1 醫療筆記與對話的 AI 轉錄
- **目標**：極大化減少臨床紀錄負擔。
- **作法**：整合大語言模型（如與本地部署的 Ollama 或雲端 API），把口述錄音或零散隨手筆記自動轉譯成結構化的 `.md` 或可視格式，歸檔在 Session 備註中。

### 3.2 療程流失預警與自動推播
- **目標**：提高復購與回診率。
- **作法**：
  - 基於 Session 間隔異常中斷、或是點數（Points）未兌換特徵，AI 模型自動圈選患者名單。
  - 與前端通知或簡訊推播功能結合，建議諮詢師進一步關懷客戶。

## 📝 綜合執行策略

1. **嚴守 Subagent 工作流**：維持「前端/後端細分 Task」＋「規格檢查／代碼品質雙重審查」體系，繼續以高紀律性完成每一項前端 UI 模組。
2. **鎖定財務一致性原則**：分潤模組已經成真，我們必須確保任何 UI 和路由的新增設計，都 **不可繞過** 現有的 `RevenueAdjustment` 審查路徑與 `locked_at` 規則。財務與事件驅動流程不可因客戶體驗妥協。
3. **優化多租戶（ClinicId）上下文**：後續前端開發必須確保全域 Store（如 Pinia 或 Context）嚴密保管與注入 `clinicId`，避免跨診所業務資料誤請求的情況發生。
