# 医生 CRM 改进计划 (Crmimprove0427)

**日期：** 2026-04-27
**状态：** 草案 / 战略提案
**研究来源：** Krayin Laravel CRM 架构分析与代码库审计

## 1. 概述
本文件概述了医生 CRM 项目的战略改进计划。该计划借鉴了高性能 CRM 架构（如 Krayin CRM）的优点，同时针对当前代码库中发现的技术债、安全风险和性能瓶颈提出了具体的改进方案。

## 2. 架构演进

### 2.1 增强模块化（受 Krayin 启发）
- **当前状态：** 虽然存在基于领域的模块，但业务逻辑往往在服务层中紧密耦合。
- **改进方向：** 采用“插件化”的模块化方法。
    - **操作：** 为模块（如 `Revenue`、`Treatments`、`Patients`）定义严格的边界。
    - **操作：** 将跨模块逻辑移动到专门的 **领域服务 (Domain Services)**，或仅通过 **事件监听器 (Event Listeners)** 处理副作用。
- **前端操作：** 拆解 `RevenueView.vue`（超过 1300 行代码），将其重构为模块化的子组件：
    - `RevenueDashboard.vue`（营收看板）
    - `RevenueRulesTable.vue`（分成规则表）
    - `RevenueRecordModal.vue`（营收记录弹窗）
    - `RevenueAdjustmentForm.vue`（调整单表单）

### 2.2 仓储模式 (Repository Pattern) 与数据访问
- **当前状态：** 使用了 TypeORM 仓储，但复杂的查询和业务逻辑经常泄露到控制器或服务中。
- **改进方向：** 严格执行仓储模式以解耦数据访问。
    - **操作：** 利用自定义仓储处理复杂的 `QueryBuilder` 逻辑，避免 N+1 查询性能瓶颈。
    - **操作：** 在所有核心实体中一致地实施“软删除 (Soft Delete)”策略。

### 2.3 事件驱动的可扩展性
- **当前状态：** 使用 NestJS `EventEmitter2` 处理通知和审计日志。
- **改进方向：** 扩展事件系统以支持外部集成和内部钩子。
    - **操作：** 为关键操作（如 `patient.created`、`treatment.completed`）实现“前置”和“后置”钩子。
    - **操作：** 确保 ACID 合规性，将触发事件的操作封装在事务中（修复 `createCourse` 中积分兑换的潜在不一致风险）。

## 3. 核心功能增强

### 3.1 动态属性（自定义字段）
- **概念：** 借鉴 Krayin CRM 的“无限自定义字段”。
- **优势：** 允许诊所在不更改数据库架构的情况下添加专门的医疗数据字段。
- **实现：**
    - 创建 `Attribute` 实体来定义字段类型（文本、日期、下拉菜单等）。
    - 创建 `AttributeValue` 实体（采用 EAV 模式或 JSONB 列）来存储患者特定的数据。

### 3.2 高级线索与漏斗管理
- **操作：** 实现患者咨询（线索）的看板式管理，类似于 Krayin 的视觉化管道。
- **操作：** 根据活动自动转换线索状态（例如，创建咨询会话时自动将线索移动到“已预约”状态）。

## 4. 安全与基础设施

### 4.1 跨租户数据隔离
- **操作：** 审计 `ClinicAuthMiddleware`。确保 JWT 中的 `user.clinicId` 始终与请求的 `clinicId` 进行校验。
- **操作：** 实现全局 `ClinicGuard`，在框架层面强制对所有标注为 `@ClinicScoped()` 的控制器进行此项检查。

### 4.2 生产环境就绪
- **数据库：** 在非开发环境中从 SQLite 迁移到 **PostgreSQL**，以处理高并发和自定义字段的 JSONB 存储。
- **数据库迁移：** 在生产环境中严格禁用 `synchronize: true`，强制执行“迁移优先”的工作流。
- **频率限制：** 集成 `@nestjs/throttler` 以保护 API 接口免受暴力破解或恶意调用。

## 5. 技术债与维护

### 5.1 重构优先级
1. **前端：** 彻底拆解 `RevenueView.vue`。
2. **后端：** 重构 `TreatmentCourseService.createCourse`，确保积分兑换逻辑的事务完整性。
3. **通用：** 在所有模块中标准化错误处理机制和 DTO 验证。

### 5.2 可扩展性
- **审计日志：** 随着 `AuditLog` 表的增长，实施日志分区或定期轮转。
- **缓存：** 引入 Redis 缓存高频访问的数据，如 `TreatmentTemplates`（疗程模板）和 `ClinicSettings`（诊所设置）。

## 6. 实施路线图
1. **第一阶段（安全与核心）：** 诊所隔离强化 + 数据库迁移策略。
2. **第二阶段（重构）：** 前端 UI 组件化 + 后端服务事务修复。
3. **第三阶段（功能）：** 自定义属性系统 + 看板式线索管道。
4. **第四阶段（性能）：** 引入缓存机制 + 审计日志优化。
