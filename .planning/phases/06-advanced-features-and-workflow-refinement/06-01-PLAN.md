# 實作計畫：Phase 6 (Hotfix) - 系統核心業務流程重構

## 1. 目標與背景 (Objective & Background)
在先前的 Phase 6 開發中，我們在 `TreatmentTemplate` (療法模板) 加入了預設的「醫令明細 (Medical Orders)」、「專屬分潤 (Revenue Rules)」以及「回診提醒間隔」。但在實際診所業務流程 (Workflow) 中，我們發現這三個設定在系統運作時存在斷層。

本計畫旨在修復這些斷層，確保「購買療程」、「執行單次操作」到「財務分潤與回診提醒」這三個核心環節完美連動。

## 2. 核心問題與解決方案 (Problems & Solutions)

### 🚨 2.1 醫令生成斷層 (Medical Order Generation Gap)
*   **問題**：患者購買療程時，系統並未將模板中的預設醫令轉化為實際的 `MedicalOrder` 實體記錄。
*   **解決方案**：修改 `TreatmentCourseService.createCourse`。在建立療程 (Treatment Course) 的同一個資料庫 Transaction 中，讀取模板的 `customMedicalOrders`，並為患者建立對應的 `MedicalOrder` 記錄。狀態設為 `pending`，次數等於療程的總堂數。

### 🚨 2.2 執行者與分潤的脫節 (Executing Staff vs Fixed Staff Gap)
*   **問題**：模板中設定的分潤是寫死在特定員工身上的。如果實際執行該次 Session 的醫師不同，錢仍會分給原設定的醫師。
*   **解決方案**：
    1. 修改 `TreatmentSession` 實體，確保記錄 `executedBy` (實際執行員工的 ID)。
    2. 修改 `RevenueCalculatorService`：當計算分潤時，如果模板設定的分潤對象是特定「角色」(如: `role:doctor`)，則系統應抓取該次 Session 的 `executedBy`，並將分潤計算給實際執行者。
    3. 前端修改：在設定模板分潤時，除了可以選擇特定「員工」，還要可以選擇「動態角色」(如：執行醫師、執行治療師)。

### 🚨 2.3 回診通知未觸發 (Follow-up Notification Gap)
*   **問題**：模板設定了回診間隔，但療程結束後系統並未發送任何通知。
*   **解決方案**：建立一個新的事件監聽器 `TreatmentEventListener`。當收到 `treatment.completed` (療程全部堂數結束) 事件時，讀取模板的 `followUpIntervalDays`，並呼叫 `NotificationService` 排程發送回診提醒。

## 3. 分階段實作計畫 (Implementation Plan)

### Step 1: 實體與前端介面調整
*   [ ] **後端**：確保 `TreatmentSession` 包含 `executedBy` 與 `executedRole`。
*   [ ] **前端**：修改 `TreatmentTemplatesView.vue`，在分潤設定的「選擇員工」下拉選單中，加入「動態：執行醫師」與「動態：執行治療師」的選項。

### Step 2: 購買療程時自動生成醫令
*   [ ] **後端**：修改 `TreatmentCourseService.createCourse`，在 Transaction 中迴圈 `template.customMedicalOrders`，建立多筆 `MedicalOrder` 記錄。

### Step 3: 分潤計算引擎升級 (認實際執行者)
*   [ ] **後端**：重構 `RevenueCalculatorService.handleCompletedSession`。
    *   讀取模板的 `customRevenueRules`。
    *   如果 `staffIdOrRole` 是 `dynamic_doctor`，則去 `Session.executedBy` 找人。
    *   確保「操作完成」的人才能分到這筆錢。

### Step 4: 療程結束自動排程回診通知
*   [ ] **後端**：建立 `TreatmentEventListener` 監聽 `treatment.completed` 事件。
*   [ ] **後端**：計算 `Date.now() + followUpIntervalDays`，並排程一筆系統/LINE通知給患者。

## 4. 驗證 (Verification)
1.  建立一個模板，包含 1 筆醫令，分潤設定為「動態：執行醫師」拿 50%。回診設定 7 天。
2.  為患者購買此療程，檢查資料庫是否成功生成 1 筆狀態為 `pending` 的醫令。
3.  由「王醫師」執行第 1 堂課並標記完成，確認系統產生 1 筆分潤記錄且對象為王醫師。
4.  將最後一堂課標記完成，確認系統成功生成一筆 7 天後的回診提醒排程記錄。
